## 3. Supabase 설정 순서 (처음 구축 시)

1. Supabase 프로젝트 생성 → Project URL, **publishable(anon) key**를 `config.js`에 입력
2. **SQL Editor**에서 아래 SQL을 **순서대로** 실행 (`supabase/` 폴더):
   1. `schema.sql` — posts/comments 등 게시판 기본 테이블
   2. `profiles-admin.sql` — profiles, admins 테이블 + RLS
   3. `profile-tax.sql` — profiles에 role/birth/address 컬럼 + tax_requests(연말정산)
   4. `profile-extra.sql` — (있다면) 프로필 추가 컬럼
   5. `community-admin.sql` — posts/comments 관리자 삭제 정책
   6. `board-extras.sql` — posts.category 컬럼 + post_reactions(반응) 테이블
   7. `withdraw.sql` — profiles 본인 삭제 정책(회원탈퇴)
   8. `resources.sql` — resources 버킷 + Storage 정책 + resources 메타테이블(자료실)
3. **관리자 등록**: 로그인 후 본인 uid 확인 → SQL Editor에서
   ```sql
   insert into public.admins (uid) values ('여기에-내-uid');
   ```
   (uid는 Supabase Authentication 화면 또는 로그인 후 콘솔에서 확인)
4. **Auth 설정**: Authentication ▸ Providers
   - Email: "Allow users without email confirmation"은 상황에 맞게
   - Kakao: 아래 4절 참고

> SQL 재실행은 모두 `drop policy if exists` / `if not exists`로 작성되어 **여러 번 실행해도 안전**합니다.

## 4. 카카오 로그인 (가장 많이 헤맨 부분)

Supabase ▸ Authentication ▸ Providers ▸ Kakao 사용.

겪은 오류와 해결:
- **KOE006 (redirect_uri mismatch)**: 카카오 개발자 콘솔의 Redirect URI에 Supabase 콜백 주소를 **정확히** 등록.
  - 콜백: `https://cetacttsdwzxjzkyozgd.supabase.co/auth/v1/callback` (프로젝트 ref `cetacttsdwzxjzkyozgd` 오타 주의 — `x` 빠뜨림 주의!)
- **KOE205 (scope not configured)**: Supabase는 카카오에 `account_email profile_image profile_nickname`을 요청함.
  - `account_email`은 **비즈 앱 전환**(사업자/고유번호증 필요)이 있어야 동의항목으로 켤 수 있음.
  - 비즈 전환이 어렵다면 `auth.js`의 `signInWithOAuth` 옵션에서 **`scopes: "profile_nickname"`** 으로 최소화(이메일 미요청).
- **키 불일치**: 카카오 "웹 로그인" 키의 **REST API 키**와 **Client Secret**을 **한 세트로** Supabase에 입력(다른 키의 시크릿과 섞으면 실패).
- 카카오 개발자 콘솔에 **웹 도메인**(GitHub Pages 주소)과 **사이트 도메인** 등록.

## 5. 핵심 설계 원칙 = 시행착오 교훈 (제일 중요)

### 5-1. supabase-js의 `getSession()`이 멈추는 문제 → **REST 직접 호출**
- 증상: 회원관리/나눔터/자료실에서 "확인 중…", "불러오는 중…"이 **무한 대기**.
- 원인: supabase-js v2의 `auth.getSession()`이 브라우저 잠금(navigator LockManager)에 걸리면 그 클라이언트로 보내는 **모든 쿼리가 함께 멈춤**.
- **해결(채택한 방식)**: SDK 쿼리를 쓰지 않고, **localStorage의 세션 토큰**을 직접 읽어 **Supabase REST API**(`/rest/v1/...`, `/storage/v1/...`)를 `fetch`로 호출.
  - 세션 읽기: `localStorage["sb-<projectref>-auth-token"]` → `access_token`, `user`
  - 모든 요청에 **8초 타임아웃**(Promise.race) + 실패 시 "다시 시도" 버튼.
  - 헤더: `apikey: <publishable key>`, `Authorization: Bearer <access_token>`
- 이 패턴은 `js/admin.js`, `js/community.js`, `js/resources.js`, `js/withdraw.js`에 동일하게 구현되어 있음(복붙해서 재사용).
- 로그인 상태 표시도 layout.js가 localStorage를 **즉시** 읽어 헤더에 반영(SDK 로딩 대기 없음).
- **로그아웃**도 `signOut()`을 기다리지 말 것(멈춤). 토큰을 먼저 지우고 즉시 이동.

### 5-2. 캐시 때문에 "바뀐 게 안 보이는" 문제
- 모든 HTML이 CSS/JS를 **`?v=YYYYMMDD<letter>`** 쿼리스트링으로 불러옴(캐시 무효화).
- **배포할 때마다 이 버전 문자열을 전부 한꺼번에 올림**(아래 배포 워크플로우의 `sed` 한 줄).
- `sw.js`는 **네트워크 전용**(앱 파일을 캐시하지 않음) — 옛 파일 잔류 방지.
- `layout.js`는 새 서비스워커가 잡히면 **자동 1회 새로고침**(controllerchange) + `reg.update()`로 옛 캐시 워커 교체.
- 그래도 안 보이면 사용자에게: **Ctrl+Shift+R**, 또는 콘솔에서 SW unregister + caches 삭제.

### 5-3. Supabase Storage 키에 한글/특수문자 불가
- 증상: 한글 파일명 업로드 시 **"Invalid key"**.
- 원인: Storage object key는 한글·`[ ]` 등 일부 문자를 허용하지 않음.
- **해결**: 파일은 **영문 안전키**(`카테고리/타임스탬프_랜덤.확장자`)로 저장하고, **원본 한글 파일명·카테고리는 `resources` 테이블**에 따로 보관해 화면에 표시. (`js/resources.js`, `supabase/resources.sql`)
- 비공개 버킷 + **서명 URL**(`/storage/v1/object/sign/...`, 1시간)로 다운로드.

### 5-4. 알림 이메일은 FormSubmit (계정/서버 불필요)
- 연말정산 신청, 새가족 등록 시 관리자 메일로 알림.
- **FormSubmit**(`https://formsubmit.co/ajax/<별칭>`)으로 전송. 계정/키 불필요.
- 최초 1회 **활성화 메일**의 'Activate Form'을 눌러야 함. 활성화는 **제출 출처(도메인) 기준** — 반드시 실제 사이트(github.io) 도메인에서 보낸 요청으로 활성화할 것.
- 실제 이메일 대신 **별칭(alias) 문자열**을 써서 공개 저장소에 이메일 노출 방지.
- (대안: Web3Forms는 이제 계정 생성 필요 → 비권장.)

### 5-5. 민감정보·헌금 원칙 (운영 정책)
- **헌금 금액은 홈페이지에 게시하지 않음**(이름만 가능). 연혁의 공사비/금액도 비공개로 정리함.
- 연말정산의 **주민번호는 뒤 1자리만** 입력받고 나머지는 별표(`3******`)로 저장.
- 주민번호 등 민감정보는 **본인+관리자만** 조회(RLS). 알림 메일에는 민감정보 미포함(이름·시각만).

### 5-6. QT(오늘의 말씀)
- 구글시트 CSV를 fetch해서 파싱. **오늘 날짜(YYYYMMDD) 이하만 표시**(미래분 숨김 — 내일 QT가 미리 보이지 않게).
- 매일 푸시는 `apps-script/qt-daily-push.gs`(Google Apps Script 트리거) + OneSignal.

## 6. 배포 워크플로우 (매번 동일)

```bash
# 0) 항상 최신화 (병렬 작업 대비)
cd "운평장로교회_홈페이지"
git fetch origin && git reset --hard origin/main   # 로컬 변경 없을 때만

# 1) 파일 수정...

# 2) 캐시 버전 일괄 올림 (예: f → g). 현재 버전 문자열을 새 문자열로:
for f in $(grep -rl "20260628f" . | grep -v "/.git/"); do sed -i 's/20260628f/20260628g/g' "$f"; done

# 3) JS 문법 점검
node --check js/main.js   # 수정한 js마다

# 4) 커밋·푸시
git add -A && git commit -m "설명" && git push origin main

# 5) 1~2분 뒤 GitHub Pages 반영 → curl로 확인 후 Ctrl+Shift+R
```

- 버전 문자열 규칙: `YYYYMMDD` + 알파벳(a,b,c…). 하루에 여러 번이면 글자만 증가.
- **JS 파일 안의 콘솔 로그 버전 표기**(`[community.js] v… REST`)도 같이 바뀜 — 어느 버전이 로드됐는지 콘솔로 확인용.
- 커밋 메시지 끝에 `Co-Authored-By: Claude …` 사용.
